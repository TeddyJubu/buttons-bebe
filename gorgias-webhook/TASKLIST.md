## The core idea (why this order works)

We merge the two options — **A) ship fast by reading policy files** and
**B) build the smart Supermemory search** — by putting a **swappable "plug"
between the draft-writer and the knowledge**: a single `kb_client.search(question)`
function.

- **Stage 1–2:** behind that function we just read Markdown files → ships fast.
- **Stage 4:** behind the *same* function we call Supermemory → smarter, with
  **zero changes to the draft engine**.

The actual words always live in a **Git repo of Markdown** (built in Stage 1);
only the *search method* over them gets upgraded. Nothing built early is thrown away.

✅ **You have a shippable product at the end of Stage 2.** Everything after makes it smarter.

Related docs: `PHASE1_KB_ARCHITECTURE.md` (KB design) · `SYSTEM_WORKFLOW.md` (full flow) · `workflow-diagram.html` (diagram).

## Stage 2 · The safe draft loop (ship it!)

Provider-abstracted OpenAI-compatible chat API (works for both Ollama Cloud and OpenRouter) via `urllib`; built-in **mock** provider for fully-offline classifier/draft testing pre-key. Public seam: `complete(messages, *, model/temperature/max_tokens/timeout/**opts) -> {text,model,usage,raw,provider}`, `complete_text(prompt, system=...) -> str`, `is_live()`, `health_check()`/`selfcheck()`; typed errors `LLMError`/`LLMConfigError`/`LLMHTTPError`. CLI `selfcheck` exits 0 in mock mode; `complete "..."`. **No real LLM key configured yet** — runs in mock mode until owner adds `llm.api_key`.
- [x] **5. `classifier.py`** — flag IMMEDIATE/sensitive tickets (refund, chargeback, dispute) so they're **escalated, not auto-drafted**. — Deterministic-first, stdlib-only rule engine (regex, word-boundary, case/whitespace-normalized). `classify(ctx_or_text, subject=None) -> Classification` accepts a `TicketContext`, a dict, or a raw string (pulls the latest `from_agent=False` message + ticket subject). Sensitive set (refund, chargeback, dispute/bank-dispute, cancellation, not_delivered/"never arrived", legal/lawyer/BBB, fraud/scam, speak-to-a-manager) -> `escalate=True`, `urgency=immediate`, `sensitive=True`, `auto_draft_allowed=False`. Benign set (shipping_status, returns_exchanges, sizing, order_change, product_question) -> draftable. Empty/garbled -> escalate (conservative bias). Optional `model_gateway` nuance pass can ONLY add caution: sensitive/escalate are OR-ed in, urgency only rises, and `auto_draft_allowed` is re-derived from `(not sensitive and not escalate)` after any LLM merge — so a rule-flagged sensitive ticket can NEVER be downgraded to auto-draftable (proven by an adversarial self-test feeding a hostile LLM opinion to a chargeback). Runs fully offline / no key. Self-test: 27 checks, prints `CLASSIFIER SELF-TEST OK`.
